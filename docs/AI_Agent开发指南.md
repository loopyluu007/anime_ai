## 📝 开发完成后的文档维护流程（重要！）

> ⚠️ **重要提示**: 每次完成开发任务后，必须立即执行以下文档同步流程，否则会导致开发进度丢失！

### 🎯 文档同步原则

1. **立即同步**: 完成开发后立即更新文档，不要等到多个功能完成后再更新
2. **全面同步**: 不仅要更新代码文档，还要更新进度追踪文档
3. **一致性检查**: 确保文档内容与实际代码实现一致
4. **版本追踪**: 每次更新都要更新文档的"最后更新"日期

## 📋 必须更新的核心文档（按优先级）

### 🔴 优先级1：进度追踪文档（每次开发后必须更新）

以下文档是追踪开发进度的核心，**每次开发后都必须更新**：

#### 1.1 `docs/02-implementation/开发进度总览.md` ⭐⭐⭐

**为什么重要**: 这是项目的整体进度总览，是了解项目状态的首要文档。

**需要更新的内容**:
1. **总体完成度百分比**
   ```markdown
   ## 📊 整体进度
   后端服务:  ████████████████████ 100%
   Web端:     ████████████████░░░░  95%  ⬆️ (如果完成相关功能)
   ```
2. **模块完成情况表格**
   - 将新完成的功能标记为 ✅
   - 更新完成度百分比
3. **更新时间线**
   ```markdown
   ### 当前阶段（2026-01-13 ~ 2026-01-20）
   - ✅ [新完成的功能]（2026-01-13）
   ```
4. **更新日志章节**
   ```markdown
   ### 2026-01-13
   - ✅ 实现[功能名称]
     - 详细描述实现的功能
   ```
5. **文档头部的"最后更新"日期**

#### 1.2 `docs/02-implementation/当前开发状态报告.md` ⭐⭐⭐

**为什么重要**: 这是最新的开发状态快照，帮助了解当前工作重点。

**需要更新的内容**:
1. **已完成的核心功能列表**
   - 添加新完成的功能到"已完成的核心功能"章节
2. **待完成的工作列表**
   - 将已完成的任务从待完成列表中移除
   - 更新任务状态（✅完成/⏳进行中）
3. **推荐的下一步任务**
   - 根据当前进度更新推荐任务
4. **文档头部的"生成时间"和"最后更新"日期**

#### 1.3 `docs/02-implementation/测试进度报告.md` ⭐⭐⭐（如果涉及测试）

**如果新增或修改了测试，需要更新**:
1. **测试覆盖率统计**
   - 更新当前测试覆盖率百分比
   - 更新各模块的覆盖率状态
2. **已完成的工作列表**
   - 添加新完成的测试用例
3. **待完成的工作列表**
   - 移除已完成的测试项

#### 1.4 `docs/05-requirements/需求和待办/TODO.md` ⭐⭐⭐

**为什么重要**: 这是任务管理文档，必须保持与实际进度同步。

**需要更新的内容**:
1. **将完成的任务移到"已完成 ✅"章节**
   ```markdown
   ## 已完成 ✅
   - [✅] [任务描述] - [完成日期]
   ```
2. **从"当前开发任务"中移除已完成的任务**
3. **更新文档头部的"更新时间"和"当前状态"**

## 🔄 文档同步标准流程（每次开发后执行）

### Step 1: 立即更新进度追踪文档（5-10分钟）

1. ✅ 更新 `docs/02-implementation/开发进度总览.md`
   - 更新总体完成度
   - 更新模块完成情况
   - 添加到更新时间线
   - 添加到更新日志
   - 更新"最后更新"日期

2. ✅ 更新 `docs/02-implementation/当前开发状态报告.md`
   - 添加到已完成功能列表
   - 从待完成列表移除
   - 更新"最后更新"日期

3. ✅ 更新 `docs/05-requirements/需求和待办/TODO.md`
   - 移动到"已完成"章节
   - 从"当前开发任务"移除
   - 更新"更新时间"

### Step 2: 根据开发内容更新代码文档（5-10分钟）

**如果开发了后端功能**:
- ✅ 更新 `backend/DEVELOPMENT.md`

**如果开发了前端功能**:
- ✅ 更新 `frontend/DEVELOPMENT.md`

**如果开发了测试**:
- ✅ 更新 `docs/02-implementation/测试进度报告.md`

### Step 3: 根据变更更新技术文档（可选，5-10分钟）

**如果添加/修改了API**:
- ✅ 更新 `docs/03-api-database/API接口设计文档.md`

**如果修改了数据库**:
- ✅ 更新 `docs/03-api-database/数据库设计文档.md`
- ✅ 更新迁移脚本

**如果修改了架构**:
- ✅ 更新 `docs/01-architecture/架构拆分方案.md`

### Step 4: 验证更新（2-3分钟）

执行以下检查清单：

- [ ] 所有进度追踪文档已更新
- [ ] 相关代码文档已更新（如适用）
- [ ] 所有更新的文档的"最后更新"日期已更新
- [ ] 文档内容与实际代码实现一致
- [ ] 已完成的任务已从待办列表中移除
- [ ] 完成度百分比已更新
- [ ] 更新日志已添加新条目

## 💡 文档同步最佳实践

### 1. 立即更新原则

✅ **推荐**: 完成功能后立即更新文档
❌ **不推荐**: 完成多个功能后再一次性更新（容易遗忘）

### 2. 原子性更新

每次更新应该是一个完整的原子操作：
- 更新所有相关的进度追踪文档
- 更新所有相关的代码文档
- 更新所有相关的技术文档

### 3. 版本一致性

确保所有文档反映的是同一版本的状态：
- 所有"最后更新"日期应该是同一天（或非常接近）
- 完成度百分比应该相互一致
- 功能状态应该在所有文档中保持一致

### 4. 详细记录

在更新日志中记录详细信息：
```markdown
### 2026-01-13
- ✅ 实现用户认证API
  - 添加登录接口 (POST /auth/login)
  - 添加注册接口 (POST /auth/register)
  - 实现JWT Token生成和验证
  - 添加单元测试（9个测试用例全部通过）
```

### 5. 格式统一

保持文档格式统一：
- 使用统一的状态标记（✅/⏳/❌）
- 使用统一的日期格式（YYYY-MM-DD）
- 使用统一的章节结构和标题格式

## ⚠️ 常见错误和避免方法

### 错误1: 只更新部分文档

❌ **错误**: 只更新了TODO文档，忘记更新开发进度总览
✅ **正确**: 同时更新所有相关的进度追踪文档

### 错误2: 忘记更新日期

❌ **错误**: 更新了内容但忘记更新"最后更新"日期
✅ **正确**: 每次更新都更新日期

### 错误3: 文档内容不一致

❌ **错误**: 开发进度总览显示功能完成，但当前开发状态报告显示进行中
✅ **正确**: 所有文档保持状态一致

### 错误4: 更新不及时

❌ **错误**: 完成功能后几天才更新文档
✅ **正确**: 完成功能后立即更新文档
